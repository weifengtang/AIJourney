# AIJourney - 记录你的 AI 编程成长之旅

> 文档版本：v1.1
> 更新时间：2026-03-27
> 项目目标：自动采集 AI 编程工具会话数据，多维度汇总成长轨迹

---

## 一、项目概述

### 1.1 核心目标
自动记录程序员与 AI 编程工具的协作过程，按时间周期自动生成工作汇总，帮助复盘成长：
- **日报**：每日工作总结，记录当天完成的工作、问题、成果
- **周报**：每周汇总，梳理本周进度、计划完成情况、下周计划
- **月报**：月度总结，分析成长曲线、技术积累、项目进展
- **年报**：年度回顾，展示全年成长轨迹

### 1.2 支持的数据来源
| 采集器 | 优先级 | 数据源 | 
|--------|--------|--------|------|
| Claude Code | P0 | `~/.claude/` JSONL 文件 | 
| CodeBuddy | P1 | `~/Library/Application Support/CodeBuddyExtension/Data` | 
| VSCode | P2 | 本地历史文件 | 需调研 |
| IDEA | P2 | JetBrains 本地数据 | 需调研 |
| 豆包网页版 | P3 | Selenium 自动化抓取 | |
| git | P4 | git 远端及本地数据 | 需调研 |
| 系统窗口 | P5 | macOS 窗口历史 | 需权限 |


### 1.3 运行方式
- 命令行：`python main.py [--date YYYY-MM-DD] [--output dir] [--daily-report-dir dir]`
- 配置：`settings.json` 中可自定义输出路径、LLM 配置
- 输出：支持 JSON（机器可读）和 Markdown（人类可读）两种格式

## 二、核心设计思想

### 2.1 插件化采集器架构
- **基类**：所有采集器继承 `BaseCollector`，必须实现 `collect()` 方法
- **插件注册**：通过装饰器 `@register_collector(name)` 自动注册新采集器
- **统一输出**：每个采集器输出标准化 `SessionData` 对象，包含会话元数据和摘要

### 2.2 数据格式
每个会话包含：
- 会话ID、来源、标题、起止时间、会话摘要

### 2.3 多级汇总架构
```
原始会话采集 → 单会话摘要 → 日报汇总 → 周报汇总 → 月报汇总 → 年报汇总
```
- 每一级都基于上一级数据重新汇总提炼
- 支持逐级下钻查看详情
- 保存所有层级的汇总结果

### 2.4 主流程
1. 按配置执行所有已启用的采集器
2. 每个采集器独立执行，失败不影响其他采集器
3. 对每个会话生成摘要（LLM 或规则提取）
4. 汇总所有采集结果，生成**日报**
5. （可选）基于多天日报生成**周报**
6. （可选）基于多周周报生成**月报**
7. （可选）基于多个月报生成**年报**

### 2.5 配置管理
- 在 `settings.json` 中可配置：
  - 输出目录 `output.output_dir`：通用输出路径
  - 日报目录 `output.daily_report_dir`：日报输出路径（可与通用输出不同）
  - LLM 参数：API 地址、密钥、模型、超时
  - 启用/禁用各个采集器
- 在 `config.py` 中硬编码默认配置

## 三、输出目录结构

### 3.1 默认结构
```
output/                     # 通用输出目录
├─ daily/                   # 日报目录（可配置）
│  ├─ YYYY-MM-DD/           # 按日期归档
│  │  ├─ 01_ClaudeCode/
│  │  │  ├─ sessions.json
│  │  │  └─ ClaudeCode会话记录.md
│  │  ├─ 02_CodeBuddy/
│  │  │  ├─ sessions.json
│  │  │  └─ CodeBuddy会话记录.md
│  │  └─ 今日工作总结_YYYYMMDD.md    # 最终日报
│  └─ ...
├─ weekly/                  # 周报目录
│  ├─ YYYY-Wxx/             # 按年份-周数归档
│  │  └─ 本周工作总结_YYYY_Wxx.md
│  └─ ...
├─ monthly/                 # 月报目录
│  ├─ YYYY-MM/              # 按年份-月份归档
│  │  └─ 月度工作总结_YYYYMM.md
│  └─ ...
└─ yearly/                  # 年报目录
   ├─ YYYY/                 # 按年份归档
   │  └─ 年度工作总结_YYYY.md
   └─ ...
```

### 3.2 配置说明
- 用户可在 `settings.json.output` 中单独配置 `daily_report_dir`
- 周报/月报/年报默认存放在 `output` 根目录对应子文件夹中
- 所有文件都按时间周期组织，便于归档和查阅

## 四、各级报告功能规划

### 4.1 日报（每日生成）
**生成时机**：每日下班/运行程序时生成
**内容要点**：
- 今日工作概览：完成了哪些任务，涉及哪些项目
- 会话列表：每个 AI 会话的目标、关键问题、成果
- 效率统计：总会话数、总 tokens 消耗、AI 辅助占比
- 遇到的问题：未解决的问题需要跟进
- 明日计划：初步规划明天要做的事情

**输出格式**：JSON + Markdown

### 4.2 周报（每周生成）
**生成时机**：每周日/周一自动汇总本周所有日报
**内容要点**：
- 本周进度概览：完成了哪些主要工作
- 按项目分类汇总：每个项目的进展
- 关键成果：本周有哪些产出
- 问题与障碍：遇到了哪些困难，需要帮助
- 下周计划：明确下周工作重点
- 成长记录：本周学到了哪些新技术

**输出格式**：Markdown（基于日报重新提炼）

### 4.3 月报（每月生成）
**生成时机**：每个月结束时自动汇总
**内容要点**：
- 月度进度总览：完成了哪些项目/功能
- 项目进展追踪：各项目的完成率
- 技术成长曲线：本月积累了哪些技术点
- 工作量统计：各类型工作占比
- 下月重点规划：明确下月优先级

**输出格式**：Markdown + 简单统计图表（文字版）

### 4.4 年报（每年生成）
**生成时机**：每年结束时自动汇总
**内容要点**：
- 年度成果总览：全年完成了哪些项目
- 技能成长轨迹：从年初到年末的技术成长
- 项目经验总结：从各个项目中学到了什么
- 下一年展望：规划下一年的技术方向和目标

**输出格式**：Markdown

## 五、设计原则

### 5.1 容错设计
- 单个采集器失败不影响整体运行
- 完整日志记录，便于排查问题

### 5.2 可配置输出路径
- 用户可在 `settings.json` 中自定义各级报告的输出位置
- 支持输出到用户指定的目录（如 Obsidian 知识库、Dropbox 同步目录等）

### 5.3 分层汇总
- 日报保留最详细信息
- 周报汇总日报要点
- 月报提炼周报核心
- 年报总结年度全景
- 每一级都保留向下钻取的可能性

### 5.4 预留扩展
- 预留性能控制、数据保留、增量采集等功能空间
- 暂不实现敏感信息脱敏，先跑通核心流程
- 支持选择只生成某一级报告
